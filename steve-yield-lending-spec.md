## 1. Overview

**Eden** is a permissionless basket/index infrastructure protocol on Base, built as a Diamond (EIP-2535) proxy. Anyone can create token baskets on Eden — single-asset vaults, multi-token indexes, or any combination. Users deposit tokens, receive receipt tokens, and earn yield from fee pot accrual and (for stEVE) epoch rewards via TWAB.

Eden is the yield engine for the $EVE ecosystem. All fees generated by baskets on Eden flow through a protocol fee split:

- **75%** → stEVE holders (protocol-level yield)
- **25%** → receipt token holders of the specific basket that generated the fee

The stEVE vault itself is a special case: 100% of its fees go to stEVE holders (no circular split).

The system includes a lending module: borrow against locked basket collateral at 0% interest with a flat ETH fee. Lending remains multi-asset like EqualIndex, and stEVE borrowers continue earning epoch rewards through internal effective-balance accounting. Default = collateral burn + principal write-off (loss isolated).

**Architecture:** Diamond proxy (EIP-2535) for upgradeability with progressive immutability — facets can be locked individually as the protocol matures.

**Lineage:** Adapted from EqualFi's `equalindex/` contracts, made standalone with no dependency on EqualFi protocol, ACI, or EqualFi's Diamond.

```

```

- NAV grows as fees accrue. Epoch rewards are emitted separately from the reward reserve and are not auto-compounded into NAV.

## 3. Core Operations

### 3.1 Mint

User deposits EVE, receives stEVE priced against current economic backing.

```
eveRequired = baseDeposit + feePotBuyIn + mintFee
```

**Fee pot buy-in:** New minters buy into the existing fee pot proportionally. This prevents value dilution to existing holders.

```
baseDeposit = (economicBalance * units) / totalSupply
feePotBuyIn = (feePot * units) / totalSupply
mintFee = (eveRequired_preFee * mintFeeBps) / 10_000
```

**First mint (totalSupply == 0):**
```
eveRequired = (bundleAmount * units) / 1e18 + mintFee
```
Where `bundleAmount = 1000e18`.

**Fee distribution on mint:**
- 10% → EVE treasury (`treasuryFeeBps = 1000`)
- Of the remaining 90%: `feePotShareBps` (e.g. 60%) → fee pot, remainder → protocol fee split (§4.2)

### 3.2 Burn

User burns stEVE, receives the fixed base bundle plus fee pot share minus burn fee.

```
gross = bundleOut + feePotShare
bundleOut = (bundleAmount * units) / 1e18
feePotShare = (feePot * units) / totalSupply
burnFee = (gross * burnFeeBps) / 10_000
payout = gross - burnFee
```

**Fee distribution on burn:**
- 10% → EVE treasury (`treasuryFeeBps = 1000`)
- Of the remaining 90%: `feePotShareBps` → fee pot, remainder → protocol fee split (§4.2)

### 3.3 Preview Functions

Required for UX and integrations:
- `previewMint(units)` → EVE required, fee breakdown
- `previewBurn(units)` → EVE returned, fee breakdown
- `nav()` → current NAV per stEVE
- `totalBacking()` → economicBalance + feePot

## 4. Fee Structure

### 4.1 Per-Basket Fees

| Parameter | Default | Cap | Authority |
|-----------|---------|-----|-----------|
| `mintFeeBps` | 100 (1%) | 1000 (10%) | Owner/Timelock |
| `burnFeeBps` | 100 (1%) | 1000 (10%) | Owner/Timelock |
| `flashFeeBps` | 50 (0.5%) | 1000 (10%) | Owner/Timelock |
| `treasuryFeeBps` | 1000 (10%) | 5000 (50%) | Owner/Timelock |
| `feePotShareBps` | 6000 (60%) | 10000 | Owner/Timelock |

**Fee waterfall on every mint/burn/flash fee:**

```
1. treasuryFeeBps (10%)  → EVE treasury (direct transfer)
2. remaining 90% split:
   - feePotShareBps (60% of remainder) → basket fee pot
   - rest → protocol fee split (§4.2)
```

Fees on mint/burn are denominated in the basket's underlying token(s). Flash loan fees follow the same denomination.

### 4.2 Protocol Fee Split

All fee pot accrual from non-stEVE baskets is split:

| Recipient | Share | Rationale |
|-----------|-------|-----------|
| stEVE holders | 75% (`protocolFeeSplitBps = 7500`) | Protocol-level yield for EVE stakers |
| Basket receipt token holders | 25% | Incentive for basket creators and holders |

**Special case — stEVE vault:** 100% of stEVE vault fees go to stEVE holders. No split (would be circular).

**Implementation:** When fees accrue to a basket's fee pot, the protocol share (75%) is routed to the stEVE fee pot. The remaining 25% stays in the basket's own fee pot for its receipt token holders.

```solidity
function _routeBasketFee(uint256 basketId, address asset, uint256 fee) internal {
    if (basketId == STEVE_BASKET_ID) {
        // stEVE vault: 100% stays in its own fee pot
        s().feePots[basketId][asset] += fee;
        return;
    }
    uint256 protocolShare = (fee * protocolFeeSplitBps) / 10_000;
    uint256 basketShare = fee - protocolShare;
    s().feePots[STEVE_BASKET_ID][_eveToken()] += protocolShare; // → stEVE holders
    s().feePots[basketId][asset] += basketShare;                // → basket holders
}
```

**Governance-controlled:** `protocolFeeSplitBps` defaults to 7500 but is updated through the owner/timelock governance path. This allows competitive adjustments (e.g., promotional rates for early basket creators) without redeployment.

| Parameter | Default | Range | Authority |
|-----------|---------|-------|-----------|
| `protocolFeeSplitBps` | 7500 (75%) | 0–10000 | Owner / Timelock |

## 5. TWAB Epoch Rewards

### 5.1 Concept

A reward reserve of **1,500,000,000 EVE** (1.5B) is distributed daily to stEVE holders over **548 epochs (1.5 years)** using a **halving decay curve**. Distribution is proportional to each holder's **time-weighted average balance** (TWAB) over each epoch.

TWAB eliminates snapshot gaming — you must hold for the full epoch to receive full weight.

### 5.1.1 Decay Curve

Emission rate halves every 183 epochs (~6 months). This front-loads rewards during the critical bootstrap phase while extending the tail.

| Period | Epochs | Daily Emission | Period Total | Cumulative % |
|--------|--------|---------------|-------------|-------------|
| Months 1–6 | 1–183 | ~4,380,000 EVE | ~801.5M | 53.4% |
| Months 7–12 | 184–365 | ~2,190,000 EVE | ~398.7M | 80.0% |
| Months 13–18 | 366–548 | ~1,095,000 EVE | ~200.4M | 93.4% |
| Residual | — | — | ~99.4M | 100% |

Residual dust (~6.6%) is retained in the reserve. Owner can allocate it to a final epoch tranche or roll it into a future rewards cycle.

**Implementation:** Each halving period stores a `rewardPerEpoch` value. On epoch transition, the contract checks which period applies:

```solidity
function rewardForEpoch(uint256 epoch) public view returns (uint256) {
    if (epoch >= totalEpochs) return 0;  // totalEpochs = 548, last paying epoch = 547
    uint256 period = epoch / halvingInterval; // halvingInterval = 183
    if (period >= maxPeriods) return 0;       // maxPeriods = 3
    return baseRewardPerEpoch >> period;      // baseRewardPerEpoch = 4_380_000e18
}
```

The owner retains `setRewardPerEpoch()` as an emergency override, but the decay schedule is the default path. Any override is capped at `baseRewardPerEpoch`, and `setRewardPerEpoch(0)` clears the override.

### 5.2 TWAB Accounting

Each user maintains a cumulative effective-balance-seconds accumulator:

```solidity
struct TwabAccount {
    uint256 cumulativeBalance;  // Σ(balance × time)
    uint256 lastUpdateTimestamp;
    uint256 lastBalance;
}
```

Rewards use internal accounting rather than `balanceOf()`. The protocol tracks:

```
effectiveBalance = liquidBalance + lockedCollateralBalance
```

On every stEVE transfer (mint, burn, transfer), the protocol updates the internal liquid-balance ledger from the transfer delta. On borrow/repay/default, it updates the locked-collateral ledger so borrowers keep accruing rewards while collateral is locked.

On every effective-balance change:
```
elapsed = block.timestamp - account.lastUpdateTimestamp
account.cumulativeBalance += account.lastBalance * elapsed
account.lastUpdateTimestamp = block.timestamp
account.lastBalance = newEffectiveBalance
```

A global TWAB tracks total effective supply the same way.

### 5.3 Epoch Structure

| Parameter | Default | Authority |
|-----------|---------|-----------|
| `epochDuration` | 86400 (24h) | Owner/Timelock |
| `halvingInterval` | 183 epochs (~6 months) | Fixed at deploy |
| `maxPeriods` | 3 | Fixed at deploy |
| `baseRewardPerEpoch` | 4,380,000 EVE | Fixed at deploy |
| `rewardReserve` | 1,500,000,000 EVE | Funded by owner |
| `totalEpochs` | 548 (~1.5 years) | Fixed at deploy |

Epochs are sequential, starting from contract deployment. No gaps.

```
currentEpoch = (block.timestamp - genesisTimestamp) / epochDuration
```

### 5.4 Reward Calculation

For a given epoch `e`:

```
userTwab_e = (user.cumulativeBalance_end - user.cumulativeBalance_start) / epochDuration
totalTwab_e = (global.cumulativeBalance_end - global.cumulativeBalance_start) / epochDuration

userReward_e = rewardPerEpoch * userTwab_e / totalTwab_e
```

If `totalTwab_e == 0`, no rewards distributed for that epoch (returned to reserve or rolled over).

### 5.5 Claiming

- `claimRewards()` — claims all pending epoch rewards for caller through the most recent completed epoch, capped at epoch `547`
- Rewards paid in raw EVE (not auto-compounded into stEVE)
- User can claim multiple past epochs in one tx
- Unclaimed rewards remain available indefinitely (no expiry)

### 5.6 Reward Reserve Management

- `fundRewards(amount)` — owner deposits EVE into reward reserve
- `setRewardPerEpoch(amount)` — emergency override (bypasses decay curve, capped at `baseRewardPerEpoch`)
- `rewardReserveBalance()` — view remaining reserve
- `currentEmissionRate()` — view current epoch's reward rate from decay curve
- When reserve is depleted, epoch rewards stop. Fee pot yield continues.
- Owner can refund reserve, but `rewardPerEpoch` can never exceed the launch `baseRewardPerEpoch`

### 5.7 TWAB Checkpoints

To enable gas-efficient historical lookups, store periodic checkpoints:

```solidity
struct Checkpoint {
    uint48 timestamp;
    uint208 cumulativeBalance;
}
```

- Checkpoint written on every balance change (mint/burn/transfer)
- Binary search for historical TWAB over any range
- Same pattern as ERC20Votes checkpoints

## 6. Lending Module

### 6.1 Overview

Borrow against locked basket collateral. 0% interest. Flat ETH fee. Fixed duration. Loss isolated on default.

Initially available for the stEVE basket (borrow EVE against stEVE). Extends to any basket in Phase 3 — borrow underlying tokens against locked receipt tokens.

Adapted from `EqualIndexLendingFacet`.

### 6.2 Loan Mechanics

**Borrow:**
```
collateralUnits = N basket receipt tokens (locked)
loanPrincipal_i = collateralUnits * bundleAmount_i * ltvBps / (1e18 * 10_000)
```

LTV is fixed at 100% of the *base bundle amount* at launch. The borrower does NOT receive the fee pot share — only the base vault backing. This means the effective LTV against NAV is always < 100% once fees have accrued.

**Repay:**
- Return exact per-asset `loanPrincipal_i`
- Receipt-token collateral unlocked

**Default (expired + unrepaid):**
- Collateral receipt tokens burned
- Per-asset `loanPrincipal_i` written off from outstanding principal tracking
- Loss isolated — other holders unaffected because redeemability invariant holds per asset

**Extend:**
- Pay flat fee again
- Maturity extended by `addedDuration`
- Cannot extend past `maxDuration` from current timestamp

### 6.3 Redeemability Invariant

The critical safety property:

```
redeemableSupply = totalSupply - lockedCollateral
vaultBalance_i >= redeemableSupply * bundleAmount_i / 1e18
```

This is checked on every borrow for every basket asset. Non-borrowers can always redeem the base bundle.

### 6.4 Lending Configuration

| Parameter | Default | Cap | Authority |
|-----------|---------|-----|-----------|
| `ltvBps` | 10000 (100%) | 10000 | Fixed |
| `minDuration` | 86400 (1 day) | — | Owner/Timelock |
| `maxDuration` | 2592000 (30 days) | — | Owner/Timelock |

### 6.5 Borrow Fee Tiers

Flat native ETH fee paid to treasury, tiered by collateral size:

```solidity
struct BorrowFeeTier {
    uint256 minCollateralUnits;  // minimum stEVE to qualify
    uint256 flatFeeNative;       // ETH fee
}
```

Tiers are ascending. Larger borrows get proportionally cheaper fees (volume discount).

Example:
| Min Collateral (stEVE) | Flat Fee (ETH) |
|------------------------|----------------|
| 1 | 0.001 |
| 100 | 0.05 |
| 1000 | 0.3 |

### 6.6 Lending Views

- `getLoan(loanId)` → loan details
- `quoteBorrow(collateralUnits)` → basket assets, per-asset principals, fee
- `maxBorrowable(asset, collateralUnits)` → max principal for that asset
- `getLockedCollateral()` → total locked stEVE
- `getOutstandingPrincipal(asset)` → total lent out for that asset

## 7. Access Control & Governance

| Role | Permissions |
|------|-------------|
| Owner / Timelock | Set fees, set reward rate, fund reserves, configure lending, pause |
| Treasury | Receives fee share + flat borrow fees |
| Users | Mint, burn, claim rewards, borrow, repay |

Owner should be a multisig or timelock for production. Single EOA acceptable for launch.

### Governance Power

stEVE is the governance token. Raw EVE carries no voting power — users must stake EVE → stEVE to participate in governance. This aligns incentives: only committed holders who are exposed to the protocol's yield and risk mechanics get a vote.

stEVE inherits ERC20Votes (via TWAB checkpoints), so voting power is snapshot-able and delegation-compatible.

Rewards and governance intentionally use different accounting:

- **Rewards:** internal `effectiveBalance = liquidBalance + lockedCollateralBalance`
- **Governance:** standard ERC20Votes balance only

This means borrowing against stEVE preserves epoch reward accrual but does not extend special governance voting treatment beyond standard ERC20Votes behavior.

## 8. Contract Architecture

Diamond proxy (EIP-2535) with facet-based modularity:

```
EdenDiamond.sol                — EIP-2535 Diamond proxy + DiamondCut
├── EdenCoreFacet.sol          — Basket creation, mint/burn, fee pot, NAV
├── EdenStEVEFacet.sol         — stEVE-specific: TWAB tracking, epoch rewards, claiming
├── EdenLendingFacet.sol       — Borrow/repay/extend/recover against basket collateral
├── EdenAdminFacet.sol         — Fee config, pause, protocol fee split, basket management
├── EdenViewFacet.sol          — Read-only: NAV, previews, basket info, lending quotes
├── EdenFlashFacet.sol         — Flash loans across any basket
└── DiamondLoupeFacet.sol      — Standard EIP-2535 introspection
```

**Receipt tokens** are standalone ERC20 + ERC20Permit contracts deployed per basket (same pattern as `IndexToken.sol`). stEVE additionally inherits ERC20Votes for governance.

### Why Diamond?

- **Upgradeability:** Add new facets (e.g., auto-compound, new basket types) without migrating state
- **Progressive immutability:** Lock individual facets as they stabilize — admin facet first, then core, then lending
- **Gas efficiency:** Shared storage across facets, no cross-contract calls for internal operations
- **Lineage:** Direct adaptation of EqualFi's proven Diamond architecture

### Permissionless Basket Creation

Basket creation requires a significant ETH fee paid to treasury, designed to deter spam and low-quality token listings. **If no creation fee is set (`basketCreationFee == 0`), permissionless creation is disabled** — only governance (owner/timelock) can create baskets.

This mirrors the pattern from `EqualIndexAdminFacetV3.createIndex()`:

```solidity
uint256 fee = s().basketCreationFee;
if (msg.sender == owner || msg.sender == timelock) {
    // Governance bypasses fee
    if (msg.value != 0) revert UnexpectedMsgValue();
} else {
    if (fee == 0) revert PermissionlessCreationDisabled();
    if (msg.value != fee) revert InvalidCreationFee(fee, msg.value);
    // fee → treasury
}
```

| Parameter | Default | Authority |
|-----------|---------|-----------|
| `basketCreationFee` | 0 (disabled) | Owner/Timelock |

**At launch:** Permissionless creation is disabled (`fee = 0`). Only governance creates baskets (stEVE is basket 0). Fee is set in Phase 3 when permissionless baskets open.

Once enabled, anyone can create a basket by calling `createBasket()`:

- Single-asset vaults (like stEVE for EVE)
- Multi-token indexes (ETH/USDC basket, meme coin basket, etc.)
- Each basket gets its own receipt token, fee pot, and vault balances
- All baskets share the Diamond's lending and flash loan infrastructure
- Basket creator can configure mint/burn fees within protocol caps
- Protocol fee split (75/25) applies automatically to all non-stEVE baskets

## 10. Security Considerations

1. **Fee pot buy-in on mint** — prevents value leakage to new entrants. Non-negotiable.
2. **Redeemability invariant** — enforced on every borrow. Non-borrowers always whole.
3. **TWAB manipulation** — flash mints don't help; balance must be held over time.
4. **Reward reserve depletion** — hard stop, no minting of unbacked rewards.
5. **Reentrancy** — all state changes before external calls. ReentrancyGuard on all mutative functions.
6. **Fee cap enforcement** — hard caps on all fee parameters, checked in setters.
7. **Loan expiry recovery** — anyone can call `recoverExpired` (permissionless liquidation). Incentive: protocol health, not profit.
8. **Native EIP-2612 permit** — gasless approvals built in. No dependency on Permit2 router.
9. **Protocol fee split routing** — stEVE basket is special-cased to avoid circular fee splits. Validated at basket creation.
10. **Diamond storage collisions** — each facet uses namespaced storage slots (`keccak256("eden.<facet>.storage")`). No shared slot conflicts.
11. **Progressive immutability** — `freezeFacet()` permanently disables `diamondCut` for a specific facet address. Irreversible. Governance-controlled.
